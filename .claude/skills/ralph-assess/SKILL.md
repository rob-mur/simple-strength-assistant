# Ralph Assess

Assess a single backlog issue and determine whether it is ready for implementation.

## Invocation

```
/ralph-assess <issue-id>
```

## Responsibilities

1. Fetch the issue (title, description, existing comments) by the given ID
2. Assess clarity
3. Assess size
4. If either check fails: tag/comment the issue with the reason, then output `STATUS: blocked <one-line reason>`
5. If both checks pass: generate a QA checklist, post it as a comment on the issue, then output `STATUS: ready`

## Step 1 — Fetch the Issue

Retrieve the issue by ID. Use whatever forge or backlog tool is available in the current environment (gh, glab, gitea CLI, forgejo CLI, curl against a REST API, etc.). You do not need to be told which one to use — figure it out from the project context.

Collect:

- Title
- Description / body
- Existing comments (to avoid duplicating a QA plan that is already there)

## Step 2 — Assess Clarity

An issue is **clear** when a developer could start implementing it without asking any questions. Ask yourself:

- Is the desired behaviour described, not just the symptom or wish?
- Are acceptance criteria present, or can they be unambiguously inferred?
- Are there no unresolved ambiguities in terminology, scope, or expected outcome?

If the issue is **not clear**, it is blocked. The blocking reason should be a single sentence naming the specific ambiguity (e.g. "blocked: acceptance criteria are missing" or "blocked: it is unclear whether this replaces or extends the existing behaviour").

## Step 3 — Assess Size

An issue is **small enough** when a single focused implementation session (one context window, no sub-tasking) can plausibly deliver it. Ask yourself:

- Does it touch a bounded part of the codebase?
- Can it be implemented and tested without needing to redesign multiple unrelated subsystems?
- Would a competent developer reasonably finish it in a few hours?

If the issue is **too large**, it is blocked. The blocking reason should name what makes it large (e.g. "blocked: issue covers both the API redesign and the UI migration — needs splitting").

## Step 4a — Blocked Path

If clarity or size failed:

1. Post a comment on the issue explaining why it cannot proceed. Be specific and actionable so the author knows exactly what to fix.
2. If the forge supports labels/tags, add a `blocked` label (or equivalent) to the issue.
3. Output (as the very last line):

```
STATUS: blocked <one-line reason>
```

## Step 4b — Ready Path

If both checks pass:

1. Generate a QA checklist (see below).
2. Post the checklist as a comment on the issue using the forge tool available in the environment.
3. Output (as the very last line):

```
STATUS: ready
```

## QA Checklist Format

The checklist must be **behavioural**: each item describes something a human tester (or automated acceptance test) can verify from the outside, without knowing or inspecting the implementation.

**Good checklist items:**

- The feature does X when the user does Y
- Submitting an invalid value shows an error message
- The existing behaviour of Z is not changed

**Bad checklist items:**

- The function `handleFoo` returns the correct value
- The database migration runs without error
- Unit tests pass

Format the comment as markdown:

```markdown
## QA Checklist

<!-- Generated by ralph-assess -->

- [ ] <behavioural check>
- [ ] <behavioural check>
- [ ] <behavioural check>
```

Aim for 3–7 items. More than 7 is usually a sign the issue is too large.

If a QA plan comment already exists on the issue (check existing comments before posting), skip posting and proceed directly to outputting `STATUS: ready`.

## Forge Agnosticism

All forge operations — fetching issues, posting comments, adding labels — are expressed as intent. Use whatever CLI, API, or project skill is available. Common examples:

- GitHub: `gh issue view <id>`, `gh issue comment <id> --body "..."`, `gh issue edit <id> --add-label blocked`
- GitLab: `glab issue view <id>`, `glab issue note create <id> --message "..."`
- Forgejo / Gitea: REST API via `curl` or a project-specific CLI
- YouTrack / Jira: REST API or project-specific skill

If you are unsure which forge is in use, inspect the git remote URL, look for a `.claude/` project skill, or check for installed CLIs.

## Output Protocol

The **very last line** of your response must be one of:

```
STATUS: ready
STATUS: blocked <one-line reason>
```

No trailing text, blank lines, or punctuation after the STATUS line.
